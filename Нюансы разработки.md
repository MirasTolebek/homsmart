# 🛠 Техническая документация: Нюансы и Баги разработки
Документ описывает неочевидные технические проблемы, решенные при создании IoT-шлюза на базе ESP32 Super Mini. Это сэкономит часы дебага при модификации кода.
### 1. Аппаратный брак антенны ESP32 Super Mini
**Симптом:** Плата не подключается к Wi-Fi (ошибка Reason: 201 - NO_AP_FOUND или AUTH_EXPIRE) на расстоянии более 10-15 сантиметров от роутера.
**Причина:** Заводской дефект разводки керамической чип-антенны (красного цвета). Сигнал теряется из-за некорректного импеданса или плохо пропаянного сглаживающего конденсатора.
**Решение:** Припайка внешней антенны (медного провода) длиной ровно **31 мм** (1/4 длины волны для 2.4 ГГц) к сигнальному (левому) контакту керамической антенны. Это восстанавливает нормальный радиус приема через несколько стен.
### 2. Зависание Inline-кнопок (Лимит памяти библиотеки)
**Симптом:** Нажатия на Inline-кнопки Telegram-бота "зависают" (крутится анимация загрузки), а плата их игнорирует.
**Причина:** Библиотека UniversalTelegramBot ограничивает входящий буфер 1500 байтами (наследие старых плат ESP8266 с малым объемом ОЗУ). При нажатии Inline-кнопки Telegram возвращает весь массив клавиатуры с эмодзи, который превышает этот лимит, и пакет отбрасывается платой на уровне библиотеки.
**Решение:** Явное расширение буфера в setup():
```cpp
bot.maxMessageLength = 4096; // Выделяем 4КБ под входящий JSON

```
### 3. Логика управления 5V-реле с 3.3V-логики ESP32
**Симптом:** Использование INPUT_PULLUP для отключения реле вызывает наводки, отвалы Wi-Fi или ложные срабатывания (особенно при запуске насоса).
**Причина:** ESP32 работает на логике 3.3V, реле на 5V. Режим INPUT_PULLUP (попытка перевести пин в высокое сопротивление) оставляет оптопару реле в "плавающем" (полуоткрытом) состоянии.
**Решение:** Строгое использование режима OUTPUT с жесткой логикой. Для реле с Low Level Trigger:
```cpp
digitalWrite(RELAY_PIN, LOW);  // Включить
digitalWrite(RELAY_PIN, HIGH); // Выключить

```
### 4. Ошибка парсинга кириллицы (UTF-8)
**Симптом:** Команда /нагрев 55 возвращает ошибку "Введите корректное число".
**Причина:** Использование text.substring(8) работает корректно только для английских слов (1 символ = 1 байт). В UTF-8 кириллица занимает 2 байта. Слово "/нагрев " весит 14 байт, поэтому жесткая обрезка по 8-му символу ломает строку.
**Решение:** Использование динамического поиска пробела для безопасного среза строки:
```cpp
int spaceIndex = text.indexOf(' '); 
float newTemp = text.substring(spaceIndex + 1).toFloat();

```
### 5. Non-Blocking архитектура (Зависания SSL)
**Симптом:** ESP32 не реагирует на температурные изменения по 6-8 секунд при активном диалоге в Telegram или накоплении сообщений.
**Причина:** Использование цикла while(numNewMessages). Каждый getUpdates() требует создания SSL-соединения и проверки сертификата (занимает ~1-1.5 сек). Если в очереди 4 сообщения, цикл while полностью блокирует основной loop() на долгое время. Датчик температуры не опрашивается.
**Решение:** Замена while на if. Плата обрабатывает строго по одной итерации Telegram-очереди за проход, оставляя процессорное время для критически важной температурной защиты.
